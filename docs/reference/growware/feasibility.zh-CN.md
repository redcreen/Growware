# Growware 可行性评估

[English](feasibility.md) | [中文](feasibility.zh-CN.md)

## 结论

Growware 作为一个反馈驱动的软件工厂或生长引擎，是可行的。

当前可行的是：分阶段、项目级、本地优先、带人工门禁的实现路线。  
当前不可行的是：跳过 `judge`、验证和部署边界，直接假设“全自动自治软件系统”已经成立。

## 这个项目到底在做什么

完整起源对话说明，Growware 真正要补的不是“再来一个 coding agent”，而是下面这层项目级控制面：

- 把 `A 窗口` 的意图、反馈和裁判标准结构化
- 把 `B 窗口` 的真实使用、日志和异常变成可处理证据
- 让 `spec / judge / code` 三类资产一起演化
- 把人类隐性判断逐步沉淀成 detector、rubric、回归测试和部署门禁

因此，Growware 不是：

- OpenClaw 的替代品
- Codex 的替代品
- 只有自动修 bug 的 daemon

它更像是附着在 OpenClaw、Codex 和目标项目之间的中间控制层。

## 为什么这个方向成立

- 原始对话已经把系统边界说清楚了：`A 窗口`、`B 窗口`、隐藏控制面，以及 `spec / judge / code` 一起演化。
- OpenClaw 适合承担 channel、gateway、plugin、hook、task 这一层生态底座。
- Codex 适合承担读代码、改代码、跑验证、输出修复结果这类受控执行任务。
- 本地优先 pilot 可以先把真实日志、工作区和部署边界接起来，再决定是否把长任务外包到云端。
- 这个项目可以先在一个真实对象上证明“造软件、修软件、学软件”三条回路，而不是一开始就追求泛化平台。

## 真正的难点

- 最大难点不是 LLM 会不会写代码，而是谁在稳定地提供“判错信号”。
- 真实环境的“正确”分成运行正确、行为正确、业务正确三层，后两层不能只靠本地 testcase。
- 如果只改 `code`，不补 `judge` 和 `spec`，系统会重复犯同类错误。
- 如果把人工反馈直接变成 patch，而不是先沉淀成规则、rubric 和回归资产，Growware 就不会真正长出学习能力。
- 一条 pilot 成功，不等于天然具备多项目、多插件并发治理能力。

## 进入实现前必须具备的条件

| 条件 | 为什么重要 |
| --- | --- |
| 一个真实 pilot 对象 | 没有真实对象，`A/B` 和运行证据都无法落地 |
| 明确的 `A 窗口` 定义 | 要清楚谁在提需求、给反馈、做审批 |
| 明确的 `B 窗口` 定义 | 要清楚哪里是真实使用，哪里会产生证据 |
| 最小 `judge` 合同 | 不先定义“什么叫错”，修复链路就不稳 |
| 验证与回归合同 | 系统需要能重复判断“这次修得是否更好” |
| 部署与回滚边界 | 没有发布安全，自动修复就是自动制造新风险 |
| 早期的人类裁判规则 | 起步阶段应该是半自动，而不是假装完全自治 |

## 对当前仓库最重要的判断

截至 `2026-04-13`，这个仓库现在最正确的动作不是写运行时代码，而是先把项目定义说准：

- 用 [origin.pdf](origin.pdf) 作为完整起源真相源
- 把公开文档从“文档先行的闭环修复项目”校正为“反馈驱动的软件工厂 / 生长引擎”
- 明确 Growware 负责的是项目级控制层，而不是重写 OpenClaw 或 Codex
- 在第一条 pilot 明确前，不锁死 daemon 形态、runtime 形态和多项目总线能力

## 当前推荐的项目姿态

1. 先本地优先，再考虑云端并行。
2. 先证明一条半自动闭环，再开放更大自治权。
3. 先把人工反馈产品化成 `judge`、规则和回归资产，再谈“系统自己长”。
4. 先做单项目或单插件 pilot，再决定是否扩成多项目平台。

## 当前讨论下的落地判断

基于目前讨论，我认为下面这组默认值是合理的：

- `Project 1 = openclaw-task-system`
- `feishu6` 作为第一阶段唯一默认的人类反馈、审批和通知入口
- `Telegram` 作为备选通道，而不是第一阶段主入口
- 所有默认挂载 `task system` 的使用 channel 视为运行面
- 项目级 durable 规则和合同写入目标项目根目录下的 `.growware/`

这组约束合理的原因是：

- 先把人类决策面收敛成单点，减少沟通面复杂度
- 让真实使用面保持开放，但不和裁判面混在一起
- 把项目级控制面和项目代码仓库对齐，便于 Git 管理和代码审查
- 把 Growware 主仓库保持为方法论和架构仓，而不是代替每个项目自己的控制面
