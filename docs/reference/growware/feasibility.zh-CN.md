# Growware 可行性评估

[English](feasibility.md) | [中文](feasibility.zh-CN.md)

## 结论

Growware 作为一个分阶段推进的工程系统，是可行的。

它现在还不具备“完全自治产品”的可行性。当前真正的缺口，不是“LLM 会不会写代码”，而是“系统有没有稳定的判错信号、合同和部署边界”。

## 为什么这个方向是现实的

- 起源对话已经给出了稳定的系统拆分：`A 窗口`、`B 窗口` 和隐藏控制面。
- 现有 coding agent 在任务边界明确时，已经很擅长读代码、分析 incident、改代码和跑验证。
- 本地优先的 pilot 能在闭环还没证实时，先把 secret、部署风险和回滚复杂度压低。
- 这个项目可以先从一个真实 pilot 起步，而不是第一天就去解决“通用自治平台”。

## 难点在哪里

- 人类期望现在主要还是自然语言，没有沉淀成机器可判定的 judge。
- 本地测试通过，并不等于真实环境正确。
- 如果没有显式验证和部署门，自动修复本身会制造新的 incident 风险。
- 跑通一个 pilot，不等于已经天然具备跨项目泛化能力。

## 进入实现前必须具备的条件

| 条件 | 为什么重要 |
| --- | --- |
| 一个具体的 pilot 目标 | 没有真实目标，系统边界就会一直停留在抽象层 |
| 清晰的 `A 窗口` 与 `B 窗口` 归属 | 反馈和运行证据不能混成一个通道 |
| 结构化 incident 合同 | 只有日志，无法稳定成为修复输入 |
| 验证合同 | 系统必须能重复判断“这个修复是否可接受” |
| 部署与回滚边界 | 没有发布安全，修复链路就不可用 |
| 早期阶段的人类审批规则 | 项目起步应该是半自动，而不是假装完全自治 |

## 建议的项目姿态

1. 先本地优先，再考虑云端执行。
2. 先证明一条半自动闭环，再考虑多目标复用。
3. 把重复的人类纠正沉淀成 durable 规则、judge 和回归资产。
4. 在第一条 pilot 明确之前，不要提前锁定 runtime 和技术栈。

## 对当前仓库的判断

截至 `2026-04-13`，这个仓库现在最正确的动作是：

- 先把起源对话以 Markdown 形式完整归档
- 把所有活跃文档统一到 `Growware` 名字下
- 在写运行时代码之前，先定义第一条 pilot loop
- 所有自治能力声明都必须放在显式的人类审批门和验证门之后
