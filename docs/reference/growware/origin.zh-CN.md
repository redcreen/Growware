## 一句话定义

Growware 是一个根据需求、使用与反馈，持续驱动软件演化的反馈驱动引擎。

如果用更贴近起源对话的表达，它也是：

- 自动反馈式的软件编写系统
- 自动反馈式的软件工厂
- 把软件“养成”而不是一次性“写完”的生长引擎

## 项目到底是什么

原始对话里，项目定义已经相当清楚：

- `A 窗口` 不是“让 AI 写代码”的聊天框，而是产品控制台
- `B 窗口` 不是“给人看结果”的界面，而是真实世界的证据源
- 后台不是一个裸 LLM，而是一套持续演化的控制面

这个控制面的职责不是只改代码，而是同时维护三样东西的一致性：

1. `spec`：软件应该做什么
2. `judge`：怎么判断它做得对不对
3. `code`：当前实现

所以 Growware 不该被描述成“闭环修 bug 工具”，也不该被描述成“文档先行的 AI coding 项目”。它更接近：

- 一套项目级演化控制系统
- 一个附着在 OpenClaw、Codex 和目标项目之间的中间控制层
- 一个把人类隐性判断逐步产品化成规则、测试和门禁的系统

## 三条核心回路

### 1. 造软件

`A 窗口` 提出意图，系统把它变成规格、边界、验收条件，再去实现、验证和部署，最后在 `B 窗口` 形成可用的软件。

### 2. 修软件

`B 窗口` 的真实使用持续产生日志、异常、失败和行为偏差。系统把这些证据转成 incident，再进入分析、修复、验证和部署。

### 3. 学软件

当 `A 窗口` 再次给反馈时，系统不应该只打一块补丁，而应该把这次反馈沉淀成以后可复用的 `judge`、规则、rubric、回归测试和不可违反约束。

这条回路解释了为什么 Growware 的目标不是“自动改代码”，而是“自动演化软件”。

## 它不该被做成什么

最容易做错的形态是：

`A 窗口一句话 -> LLM 直接改代码 -> 部署`

这条链路短，但非常不稳。完整起源对话里反复强调，真正不能省的是中间的 `judge layer`。

更稳的方式是：

`A 窗口反馈 -> 先更新 spec / rubric / detector / eval -> 再改代码 -> 再验证 -> 再部署`

也就是先教系统“什么叫错”，再让系统去修。

## Growware 这个名字为什么成立

原始对话里，`auto-software` 被判断为更像方向名或概念名，而不是最终的系统名。

`Growware` 最后成立，是因为它比 `grow software` 更像一个独立系统名，同时又保留了“软件不是写完的，而是在使用与反馈中长出来”的核心气质。

对应的中文表达里，最贴近的系统感说法是：

- 生长引擎
- 根据需求、使用与反馈，持续驱动软件演化的核心系统

## 对当前仓库文档的直接约束

基于完整起源对话，仓库文档后续必须遵守这些边界：

- 不能把 Growware 写成单纯的 AI coding assistant
- 不能把 Growware 写成只会自动修 bug 的守护脚本
- 必须保留 `A 窗口 / B 窗口 / 隐藏控制面` 这个系统级拆分
- 必须把 `spec / judge / code` 视为共同演化对象，而不是只写 `code`
- 第一阶段可以先做半自动和本地优先，但项目定义本身不能被说窄
